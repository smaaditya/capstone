Abstract  
Quantum computers aren't just a theoretical concern anymore, they're becoming real, and 
that's a problem for our current security systems. The RSA and ECC algorithms  that we 
rely on today; Shor's algorithm can break them efficiently once we have sufficiently 
powerful quantum machines. This realization is what led toward exploring post -quantum 
cryptography.  
 
This project deals with implementing CRYSTALS -Kyber, which NIST finally 
standardized in 2024 after years of evaluation. We built a working system that uses Kyber 
for key exchange combined with AES -256 for actually encrypting image files. The setup 
has two parts: a host application running on a regular computer (Windows with MSYS2), 
and a receiver module designed for RISC -V processors that can run on actual FPGA 
hardware or just simulation.  
 
The implementation handles the complete workflow, which consists of generating keys, 
encapsulating them, encrypting images, and decrypting them back. We tested it with 
different image sizes and formats to see if everything works correctly. The results show  
that while Kyber adds some overhead compared to traditional methods, it's manageable and 
gives us security that should hold up even against quantum attacks.  
 
For embedded systems like IoT devices or secure communication modules, this kind of 
implementation matters because we can't just upgrade hardware easily once deployed. 
Getting this working on RISC -V specifically is useful since it's an open architecture ga ining 
traction in embedded spaces.

 CHAPTER 1  
Introduction  
 
At present, the world of cybersecurity faces a “cryptographic cliff.” Over the past few 
decades, the security of practically all monetary transactions on the Internet, confidential 
government communication, and private information has depended on the assum ption of 
the difficulty of performing integer factorization (RSA) and discrete logarithm 
computations (ECC). The validity of relying on cryptography has already been questioned 
by the algorithm proposed by Peter Shor since as early as 1994.  
 
This algorithm showed that an appropriately capable quantum computer, based on qubits 
and quantum superposition principle, could perform the required computations in 
polynomial time. In other words, whereas a classical computer would require billions of 
years to break a 2048 -bit RSA key, a quantum computer would only need minutes to do so. 
Due to the problem posed by such a development, the idea of "Harvest Now, Decrypt Later" 
appeared.  
 
As a result of this fundamental challenge to conventional cryptography, the US National 
Institute of Standards and Technology (NIST) launched a worldwide contest in 2016 for 
the creation of new standards in Post -Quantum Cryptography (PQC). In an almost dec ade-
long process of cryptanalysis, August 2024 saw a landmark moment in the adoption of 
CRYSTALS -Kyber into a standard named FIPS 203.  
 
CRYSTALS -Kyber utilizes the Module Learning With Errors (M -LWE) problem. While 
conventional cryptography such as RSA relies on number theory principles, M -LWE is 
based on the geometric structure of lattices. This means that the problem revolves around 
searching for a "secret" vector in a multi -dimensional space that has had noise injected into 
it. Using even the most advanced quantum -accelerated algorithms makes it impossible to 
navigate the lattice space.  
 
The move from theoretical to practical is even harder when considering embedded systems. 
Systems employed in the Internet of Things (IoT), industrial control, and edge computing 
often have very constrained hardware resources, such as reduced SRAM, slower C PU clock 
rates, and smaller energy footprints.

Existing Kyber designs require significant amounts of memory since public keys and 
polynomial matrices are rather large in size. Moreover, many embedded devices have been 
placed in "set -and-forget" locations (e.g., components on satellites, underwater sens ors) 
where any hardware changes cannot occur. Thus, our project aims at addressing these 
challenges by showing the feasibility of using post -quantum cryptography on a low -power 
RISC -V architecture.  
 
For this task, a highly advanced Key Encapsulation Mechanism (KEM) is applied in 
conjunction with Data Encapsulation Mechanism (DEM). Hybrid model architecture is 
used in this solution, which involves:  
 
Asymmetric Encryption Stage: Key encapsulation mechanism (Kyber -768) is employed in 
the early stages of the encryption process. In this way, even if a third party manages to 
intercept the transmission, they won’t be able to decipher the key using quantum 
computing.  
 
Symmetric Encryption Stage: AES -256-GCM (Galois/Counter Mode) is used once the 
encryption key is successfully encapsulated. AES -256 is used due to its resistance to 
quantum computing Grover's Algorithm.  
 
The architecture mimics an existing "Infrastructure -to-Edge" setup.  
 
The Sender (Host): Simulates a powerful server/gateway that handles the complex 
operations of Kyber encapsulation and AES encryption.  
 
The Receiver (RISC -V): Simulates the edge device that undertakes the resource -intensive 
tasks of decapsulation and decryption.  
 
Two different communication approaches are used in the project: a file -based protocol for 
offline testing and a Memory -Mapped UART (Universal Asynchronous Receiver -
Transmitter). The latter is an important technical achievement since it enables the RISC -V

 CHAPTER 2  
Literature Survey  
 
Quantum resistance is no longer an abstract cryptographic concept but an established part 
of security engineering. In particular, NIST has recently completed the creation of the core 
standards for lattice -based protocols for key encapsulation and digital s ignatures: FIPS 203 
ML-KEM (formerly Kyber) [1] and FIPS 204 ML -DSA (formerly Dilithium) [3], 
respectively. ML -KEM and ML -DSA are based on solving the shortest vector problem in 
structured lattices, a mathematically difficult problem unsolved by any of the  known 
quantum or classical algorithms (e.g., BKZ algorithm) [13][15].  
 
2.1 The RISC -V Architecture: An Open, Modifiable Architecture for Encryption  
The development of RISC -V has been revolutionary for the semiconductor sector since it 
offers an open source ISA as opposed to proprietary x86 or ARM architectures [7]. What 
makes this architecture so powerful in relation to cryptography is its modularity.  Unlike 
fixed architectures where a number of features need to be included during design time, 
RISC -V can allow developers to first implement a base integer architecture (e.g., RV32I, 
RV64I) and add certain extensions, such as 'M' (integer multiplication) or 'K' (scalar 
cryptography) [18].  
 
Research has shown that despite the fact that RISC -V chips might currently have lower 
clock frequencies compared to specialized x86 or ARM devices, they are much better at 
IPC in memory bound kernels [10]. This property of the architecture is especially 
important for the Hornet core and others that try to maximize encryption performance by 
milliwatt [4][11].  
 
2.2 Algorithmic Limitations: Keccak, SHAKE, and NTT  
The computational cost of CRYSTALS -Kyber is primarily incurred in three major 
operations: hashing, extendable output functions, and polynomial multiplication 
[16].2.2.1 Keccak and SHAKE (FIPS 202)The permutation operation of Keccak -p[1600, 
24] underlies the hash algorithms SHA3 -256 and SHA3 -512, and also the extendable -

5 
 output functions (XOF) named SHAKE -128 and SHAKE -256 [2]. The SHAKE algorithm 
in Kyber is widely employed for "Sponge Construction," which is responsible for 
generating the large pseudo -random polynomial matrices ($A$) based on a tiny seed 
reference implem entation has proven inefficient in dealing with the 24 rounds of 
permutations demanded by the Keccak operation. However, research on 64 -bit dual -issue 
RISC -V systems indicates that through smart scheduling of the "Rho" and "Pi" stages of 
the permutation in  parallel in two pipelines, software engineers can obtain a 59.18% 
performance enhancement relative to the regular reference implementation [5]. This 
algorithmic improvement is critical for image encryption, where SHAKE needs to process 
thousands of bytes to generate key expansions for every frame of images.  
 
2.2.2 Number Theoretic Transform (NTT)  
The most computationally complex operation within Kyber is polynomial multiplication. 
This is carried out with O(n^2) complexity within the time domain. In order to attain the 
necessary O(n \log n) complexity for a real -time system, Kyber makes use of a sp ecial 
kind of FFT called the Number Theoretic Transform (NTT) [12].NTT performance on 
RISC -V is largely influenced by modular reduction, namely Montgomery and Plantard 
arithmetic. These two methods enable the processor to carry out lazy reduction while 
avoiding costly divisions within the module. RISC -V assembly optimized for modular 
multiplication takes advantage of the 'M' extension to accomplish the butterfly 
computation of the NTT in fewer cycles, decreasing the decapsulation process latency for 
process ors such as BOOM or Hornet [22][23].  
 
2.3 Parallelism at the Hardware Level: SIMD & Dual -Issue Pipeline  
Two important aspects of architecture have been employed to overcome the performance 
disparity between software and hardware:  
Packed -SIMD (P -Extension): Cryptography using lattices requires operations on numbers 
of relatively small size (coefficients). The P -extension of the RISC -V instruction set 
allows multiple 16 -bit/8-bit integers to be operated using a single instruction in a 32-
bit/64 -bit register [19]. The use of "vector in scalar" gives large gains in terms of speed in 
terms of polynomial addition and scaling, which happen often in the Kyber pipeline [20].

 Dual -Issue Pipeline: By employing CPUs capable of fetching and executing two 
instructions at once, the memory access overhead is reduced significantly. Particularly, 
while one pipeline carries out the mathematical calculation of a matrix -vector 
multiplicat ion operation, the other can pre -fetch the next row of the image to be encrypted 
[11].  
2.4 Evolution of Standards: From FIPS 202 to FIPS 203  
The roadmap of Post -Quantum Cryptography began with the formalization of SHA -3 
(FIPS 202) [2], which introduced the industry to the sponge -based hash design. This was 
followed by the NIST PQC competition, which resulted in the selection of Kyber and 
Dilith ium. 
Current research, such as the ECO -CRYSTALS project, represents the pinnacle of this 
evolution. It is the first highly -optimized assembly -level implementation to deploy these 
NIST winners on the 64 -bit RISC -V ISA. These projects prove that by combining the 
mathematical elegance of ML -KEM with the architectural flexibility of RISC -V, we can 
achieve a security posture that is not only quantum -resistant but also efficient enough for 
the next generation of embedded industrial and medical devices [6][25].

3.1 System Architecture & Design Approach  
The architectural basis of this secure image transmission system is the Heterogeneous 
Client -Server Approach. This approach acknowledges the pragmatic nature of the current -
day IoT and industrial environment, where data may originate from or reside in powe rful 
"Host" hardware systems but need to be securely consumed by resource -starved "Edge" 
computing platforms. For this project, the Sender (Host) is the high -end x86_64 computer 
running inside an MSYS2 environment, whereas the Receiver (Target) is the cust om Hornet 
RISC -V processor.  
The system follows a Three -Layer Logical Approach:  
Application Layer:  This layer deals with high -level data management. For the Sender, it 
includes handling image file parsing from standard formats like PNG/JPG to raw pixel 
arrays and constructing packets. For the Receiver, it involves allocating memory for the 
input buffer  and ultimately reconstructing the pixel data.  
The Cryptographic Layer:  This forms the heart of the project. The design incorporates 
the CRYSTALS -Kyber (ML -KEM) protocol for quantum -resistant key exchange and 
AES -128-GCM for fast symmetric encryption of data. With the separation of the key 
exchange from the data encryption, t he "Hybrid Security," which means quantum resistance 
for key exchange and silicon proven efficiency for the bulk data, is achieved.  
 
The design of Kyber -768 is per the FIPS 203 standard and is based on the Module Learning 
With Error (M -LWE) problem. While the security in the case of RSA is based on 
factorization of prime numbers, the security in the case of Kyber is based on the hardnes s 
of finding the shortest vector of the high dimensional lattice.  
The secure image transmission system uses a client -server model with an asymmetric 
architecture, wherein the server sender is a general -purpose computer having sufficient 
computing resources, whereas the secure receiver is designed for platforms with limit ed 
resources (RISC -V-based computing environment).

3.2 Kyber Cryptographic Implementation: The Lattice Engine  
3.2.1 The Number Theoretic Transform (NTT) and Butterfly Logic  
At the heart of Kyber is the multiplication of polynomials in the ring R_q = {Z}_q[X]/(X^n 
+ 1), where  n=256 and q=3329. A standard multiplication would require O(n^2) operations, 
which is prohibitively slow for a RISC -V core. To optimize this, we utilize the Number 
Theoretic Transform (NTT)  to move polynomials into the "Frequency Domain," reducing 
complexity to O(n \log n).  
The transform uses the Cooley -Tukey (CT) Butterfly  for the forward transform and the 
Gentleman -Sande (GS) Butterfly  for the inverse transform. In a CT butterfly, we take 
two coefficients and a "twiddle factor" (a power of the root of unity ) to produce two new 
values. This allows for "In -Place" computation, meaning the 256 coefficients are updated 
within the same memory block, drastically reducing the SRAM footprint on the Hornet 
core. The inverse NTT (iNTT) then uses the GS butterfly to move the data back t o the 
"Time Domain" after point -wise multiplication is complete.  
3.2.2 Modular Reductions: Montgomery and Barret  
Since all calculations in the Kyber encryption algorithm must be done modulo q=3329, the 
processor will continuously have to do reductions. Using the % operator on a RISC -V 32 -
bit architecture that does not have a hardware divider is very slow. Therefore, we 
implement the Montgomery reduction method. The Montgomery reduction method 
converts the modular reduction problem into a problem of multiplication and bit shifting 
by converting numbers into the "Montgomery Domain" (by scaling by R=2^16). 
Furthermore, B arret reduction is utilized for small constant -time reductions during 
polynomial additions to prevent coefficient overflow.  
3.3 The Hashing and Randomness Core: Keccak and SHAKE  
Keccak -p[1600] permutation is the most computationally expensive module in the Kyber 
stack and enables the use of the SHA3 -256/512 hash function family and the SHAKE -
128/256 Extendable -Output Functions (XOF).3.3.1 The Keccak Sponge 
Construction"Sponge Cons truction" is the basic idea behind the Keccak permutation, which

13 
 comprises two steps: absorb and squeeze. In the former, the input message (e.g., the seed 
of the public key) is XORed into the internal 1600 -bit state, which is subjected to 24 rounds 
of Keccak -f permutation. In the latter stage, pseudo -random bits are "sq ueezed" out of the 
internal state.In your particular project, SHAKE -128 function acts as a Pseudo -Random 
Function (PRF) and expands the initial 32 bytes long seed to produce the giant matrix $A$ 
used in the public key generation process. To economize memor y, this process is carried 
out "on -the-fly," so no full matrix is saved beforehand, but rather, all necessary coefficients 
are computed only when they are needed in the NTT multiplication procedure. Such 
approach, called "Rejection Sampling," allows genera ting uniformly distributed 
coefficients in the range from 0 to q -1. 
 
Kyber key generation follows the FIPS 203 specification with the following procedural 
steps:  
1. Generate uniform random seeds ‘d’ and ‘z’, each 32 bytes  
2. Expand seed ‘d’ into pseudorandom matrix A ∈ R_q^(k×k) using SHAKE -128 in 
counter mode  
3. Sample secret vector ‘s’ and error vector ‘e’ from centered binomial distribution 
CBD_η  
4. Compute public vector ‘t = A·s + e ’ in the polynomial ring R_q  
5. Encode public key pk = (t mod^+ q, d) and secret key sk = (s, e, tr, z) with tr = H(t)

3.2.2 Encapsulation and Decapsulation  
Encapsulation  Algorithm  derives shared secret and ciphertext:  
 
1. Generate random message m ∈ {0,1}^256  
2. Compute ( K̄, r) = G(m ‖ H(pk)) where G and H are hash functions  
3. Sample error vectors e1, e2 from CBD_η using randomness r  
4. Compute u = A^T·r  + e1 and v = t^T·r + e2 + Decompress_q(Encode_1(m), 1)  
5. Compress and encode ciphertext c = (Compress_q(u, d_u), Compress_q(v, d_v))  
6. Derive shared secret K = KDF( K̄ ‖ H(c))  
 
Decapsulation  Algorithm recovers the shared secret:  
1. Decompress ciphertext to obtain u and v  
2. Compute m' = Encode_1(Compress_q(v - s^T·u, 1))  
3. Recompute ( K̄', r') = G(m' ‖ H(pk))  
4. Verify ciphertext validity through re -encryption; if invalid, set K̄' to random z value 
(implicit rejection)  
5. Output K = KDF( K̄' ‖ H(c))  
 
The Number Theoretic Transform accelerates polynomial multiplication, reducing O(n²) 
complexity to O(n log n) through decomposition into the frequency domain.

3.4 AES -128-GCM: Secure Data Encapsulation  
3.4.1 Counter Mode (CTR) and Galois Authentication  
The encryption algorithm used is the Galois/Counter Mode (GCM). This encryption 
algorithm involves CTR encryption and Galois Message Authentication Codes (GMAC) 
encryption.  
Encryption:  The AES engine takes in an incrementing value which is the combination of 
an initial vector and counter and then encrypts it by using the XOR method with the raw 
image pixel values. This results in a stream cipher mode of the block cipher. It enables any 
data length irrespective of padding methods.  
Authenticity (The GHASH):  The 128 -bit authentication tag is computed from a universal 
hash function (GHASH) in the Galois field GF(2^{128}).  
Verification:  After receiving the image, the RISC -V processor will recalculate the tag. 
Should there be a corruption of any bit in the image as a result of the UART transfer, the 
tags will not be equal, hence the receiver discards the information. This gives us the

3.5 Communication Protocol for Hardware Integration  
The communication protocol follows the principle of "Minimalist Robustness". This 
includes using a structured frame that would inform the RISC -V processor of the incoming 
data.  
3.5.1 UART Memory -Mapped I/O (MMIO)  
UART hardware is mapped to an address space of Hornet (for example, 0x10000000). The 
software interfaces with the UART hardware via writing and reading from this address.  
Transmitter (Host): Data sent is in the following order: [Header: 4 bytes] [Kyber 
Ciphertext: 1088 bytes] [AES -IV: 12 bytes] [Encrypted Image: N bytes] [Auth Tag: 16 
bytes].  
Receiver (RISC -V): An I/O loop with a Polling approach is used. It checks the "Line Status 
Register" to find out if a byte is available. After this byte becomes available, it is read into 
a small FIFO buffer, then decrypted by AES engine and finally stored  in an "Image Result" 
memory block.

4.2 Physical design  
To address resource constraints encountered during development, we initially attempted to 
validate the Kyber -768 and AES -128-CTR pipeline on a physical FPGA. However, the 
design’s high computational density and memory requirements for post -quantum 
cryptogr aphic primitives exceeded the available hardware area. Consequently, the project 
transitioned to a full physical design flow. This approach allowed for granular optimization 
of the Hornet RISC -V core and dedicated crypto -accelerators, ensuring the complex 
arithmetic logic fit within a tailored silicon floorplan. Shifting to an ASIC -oriented physical 
design enabled us to maintain high performance while satisfying the strict area and timing 
constraints inherent in high -security image encryption pipelines.

4.3 Security Analyses  
 
Kyber -768 provides NIST Security Level 3, indicating resistance comparable to AES -192 
against both classical and quantum adversaries. The underlying MLWE problem has 
withstood extensive cryptanalytic investigation.  
Implementation Security : The reference implementation incorporates:  
• Constant -time polynomial arithmetic preventing timing attacks  
• Implicit rejection preventing chosen -ciphertext attacks  
• Proper randomness generation for key material  
Identified Limitations : 
1. Side-Channel Resistance : Power analysis and electromagnetic side -channel 
countermeasures are not implemented. Production deployments requiring resistance 
to sophisticated attacks would need additional protections.  
2. Memory Protection : While this clears sensitive buffers, comprehensive protection 
against cold boot attacks or memory forensics would require additional operating 
system or hardware support.

 3. Key Authentication : The protocol assumes authentic public key distribution. Man -
in-the-middle attacks are possible without additional authentication mechanisms 
(certificates, pre -shared keys, etc.).  
4. Error Handling : Current implementation exits on most errors; production systems 
would require robust error recovery and logging.  
Despite these limitations appropriate to a proof -of-concept implementation, the core 
cryptographic operations are correct and match NIST test vectors.

void uart_write (void *uart_base , unsigned  char *data, size_t len) { 
    volatile  unsigned  int *tx_ctrl = (unsigned  int *)((char 
*)uart_base  + UART_TX_CTRL ); 
    volatile  unsigned  int *tx_data = (unsigned  int *)((char

    for (size_t i = 0; i < len; i++) {

    } 
}

    }
